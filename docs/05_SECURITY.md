## 1. セキュリティ原則

### 1.1 基本方針

TalentSyncは以下のセキュリティ原則に基づいて設計・運用されています。

| 原則 | 説明 |
|------|------|
| **最小権限の原則** | 必要最小限のアクセス権限のみを付与 |
| **多層防御** | 複数のセキュリティ層で保護 |
| **セキュアバイデザイン** | 設計段階からセキュリティを考慮 |
| **ゼロトラスト** | すべてのアクセスを検証 |
| **透明性** | セキュリティ対策を顧客に開示 |

## 2. インフラストラクチャセキュリティ

### 2.1 ホスティング

| 項目 | 詳細 |
|------|------|
| **フロントエンド** | Vercel（グローバルCDN） |
| **バックエンド** | Google Cloud Run (asia-northeast1) |
| **データベース** | Neon serverless PostgreSQL |
| **コンテナ** | Docker (Node 20-slim、非rootユーザー実行) |

### 2.2 ネットワークセキュリティ

| 対策 | 説明 |
|------|------|
| **TLS暗号化** | すべての通信はTLS 1.3で暗号化 |
| **DDoS保護** | Vercel / Cloud Run による自動防御 |
| **CORS制限** | ホワイトリスト方式のオリジン検証 |
| **エッジネットワーク** | Vercel CDNによるフロントエンド配信 |

### 2.3 証明書管理

- 自動証明書発行・更新（Vercel / Cloud Run 管理）
- HSTS（HTTP Strict Transport Security）有効

## 3. アプリケーションセキュリティ

### 3.1 OWASP Top 10対策

| 脆弱性 | 対策 |
|--------|------|
| **インジェクション** | Drizzle ORM のパラメータ化クエリ、Zod バリデーション |
| **認証の不備** | Better Auth セッション管理、HttpOnly Cookie |
| **機微データの露出** | 暗号化、最小権限アクセス、RBAC |
| **XXE** | XMLパーサー無効化 |
| **アクセス制御の不備** | ミドルウェアによるRBAC、ルートごとの権限チェック |
| **セキュリティ設定ミス** | セキュリティヘッダー自動設定 |
| **XSS** | フロントエンドでの DOMPurify サニタイズ、CSP ヘッダー |
| **安全でないデシリアライゼーション** | Zod による型チェック・バリデーション |
| **既知の脆弱性** | 定期的な依存関係更新（Dependabot） |
| **不十分なログ** | X-Request-Id によるリクエストトレーシング |

### 3.2 入力バリデーション

```
クライアント側バリデーション（Vue 3 リアクティブ）
        ↓
APIバリデーション（Zod スキーマ検証: 型、長さ、形式）
        ↓
サニタイズ処理（DOMPurify）
        ↓
パラメータ化クエリ（Drizzle ORM）
        ↓
データベース保存
```

### 3.3 セキュリティヘッダー

| ヘッダー | 値 |
|----------|-----|
| `X-Content-Type-Options` | nosniff |
| `X-Frame-Options` | DENY |
| `Strict-Transport-Security` | max-age=31536000; includeSubDomains |
| `Content-Security-Policy` | default-src 'self' |

### 3.4 レート制限

| 対象 | 制限 | 実装方式 |
|------|------|----------|
| 一般エンドポイント | 100リクエスト/分 | インメモリ（IPベース） |
| 認証エンドポイント | 10リクエスト/分 | インメモリ（IPベース） |

### 3.5 AIプロンプトインジェクション対策

- ユーザーデータ内のMarkdown文字列のサニタイズ
- システムプロンプトによるロール変更指示の拒否
- コンテキストデータの整形・最小限化

### 3.6 依存関係管理

- 週次での脆弱性スキャン（npm audit）
- Dependabotによる自動更新PR
- 重大な脆弱性は24時間以内に対応

## 4. データ保護

### 4.1 データ分類

| 分類 | 例 | 保護レベル |
|------|-----|-----------|
| **機密** | 評価コメント、1on1内容 | 暗号化＋アクセス制限 |
| **社外秘** | SF結果、SPI結果 | アクセス制限 |
| **社内** | 氏名、所属チーム | 認証必須 |

### 4.2 暗号化

| 対象 | 方式 |
|------|------|
| **通信** | TLS 1.3 |
| **保存データ** | Neon PostgreSQL のストレージ暗号化 |
| **セッション** | Better Auth によるシークレットキー暗号化 |

### 4.3 データ保持

| データ種別 | 保持期間 |
|------------|----------|
| アクティブデータ | 契約期間中 |
| 退職者データ | 退職後5年 |
| 監査ログ | 3年 |
| バックアップ | 30日 |

### 4.4 データ削除

契約終了時のデータ削除プロセス：

1. 顧客にデータエクスポート機会を提供（30日間）
2. 本番データの完全削除
3. バックアップからの削除（30日以内）
4. 削除完了証明書の発行

## 6. 監査とログ

### 6.1 リクエストトレーシング

各リクエストに UUID（X-Request-Id）を付与し、処理フローを追跡可能。

### 6.2 監査ログ

以下の操作を記録：

| カテゴリ | 操作例 |
|----------|--------|
| 認証 | ログイン、ログアウト、パスワード変更 |
| データ変更 | 作成、更新、削除 |
| 権限変更 | 権限付与、権限削除 |

### 6.3 ログ項目

| 項目 | 説明 |
|------|------|
| タイムスタンプ | UTC形式 |
| リクエストID | X-Request-Id (UUID) |
| ユーザーID | 操作者の識別子 |
| IPアドレス | 接続元IP（x-forwarded-for対応） |
| 操作種別 | HTTP メソッド |
| 対象リソース | エンドポイントパス |

### 6.4 ログ保管

- 保管期間: 3年
- アクセス制限: システム管理者のみ

## 7. インシデント対応

### 7.1 インシデント分類

| レベル | 定義 | 対応時間 |
|--------|------|----------|
| **Critical** | データ漏洩、サービス停止 | 1時間以内 |
| **High** | セキュリティ侵害の疑い | 4時間以内 |
| **Medium** | 脆弱性発見 | 24時間以内 |
| **Low** | 軽微なセキュリティ問題 | 72時間以内 |

### 7.2 対応プロセス

```
検知 → 初期対応 → 調査 → 封じ込め → 復旧 → 報告 → 改善
```

### 7.3 顧客への通知

- Critical/Highインシデント: 24時間以内に通知
- 影響を受けた顧客への個別連絡
- インシデントレポートの提供
